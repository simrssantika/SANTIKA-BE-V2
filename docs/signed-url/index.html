Signed URL

Utilitas signing global berbasis HMAC-SHA256 di package global/signedurl. Bisa dipakai semua resource & service — bukan khusus file. Validasinya independen dari user/JWT: cukup berbasis shared secret, sehingga cocok untuk transaksi berat lintas service yang butuh validasi.

Kapan dipakai?

Use caseContoh
Validasi lintas serviceSpring Boot ↔ Go PDF service ↔ Gotenberg — request callback divalidasi tanpa JWT user
Akses fileEmbed foto pegawai di <img src>, link download sementara
Tanda tangan payloadSign transaction id / body antar service untuk cegah tampering
ℹ
Independen dari user. Tidak butuh JWT, tidak terikat sesi login. Yang dipercaya hanya secret bersama (app.signed-url.secret). Inilah yang membuatnya bisa dipakai service-to-service.

Dua Mode

1. Signed URL

signUrl(path, ttl) · verifyUrl(path, expires, sig)

Menempelkan ?expires=&signature= ke sebuah URI. Untuk akses resource via link berbatas waktu.

2. Raw HMAC

sign(payload) · verify(payload, sig)

Tanda tangan string/payload bebas. Untuk validasi body request, transaction id, atau callback antar service.

Cara Kerja

1. Pihak A menandatangani

Service memanggil signUrl(path) atau sign(payload) dengan secret bersama.

2. Signature dikirim

Lewat query URL (?expires=&signature=) atau header/body untuk payload.

3. Pihak B memvalidasi

Untuk URL: SignedUrlInterceptor global validasi otomatis. Untuk payload: panggil verify(). Tanpa perlu JWT.

Dari Service (Kotlin)

Inject SignedUrlService di service mana pun.

import com.santika.simrs.global.signedurl.SignedUrlService

@Service
class PegawaiService(
    private val signedUrlService: SignedUrlService
) {
    fun detailWithFoto(id: UUID): PegawaiDetailRes {
        val pegawai = ...
        // Sisipkan signed URL foto, berlaku 10 menit
        val foto = signedUrlService.signUrl(
            "/api/v1/files/signed/storage/${'$'}{pegawai.fotoId}", ttlSeconds = 600
        )
        return pegawai.copy(fotoUrl = foto.url)
    }
}

Validasi Lintas Service

Skenario: Spring Boot memicu Go PDF service untuk render dokumen. Go service perlu memanggil balik (callback) atau mengakses data, dan Spring perlu memastikan request itu sah — tanpa menyertakan JWT user.

Mode URL — Go service akses data

// Spring: buat signed URL untuk Go service
val dataUrl = signedUrlService.signUrl(
    "/api/v1/laporan/${'$'}{laporanId}/data", ttlSeconds = 120
)
// kirim dataUrl.url ke Go service → Go fetch URL itu
// SignedUrlInterceptor di Spring otomatis memvalidasi saat Go memanggilnya

Mode Raw — tanda tangan payload callback

// Spring: tanda tangani transaction id sebelum kirim ke Go
val sig = signedUrlService.sign("trx:${'$'}{trxId}:render")
// Go mengirim balik trxId + sig saat callback

// Spring: validasi callback dari Go
if (!signedUrlService.verify("trx:${'$'}{trxId}:render", sig))
    throw UnauthorizedException("Callback tidak sah")
✓ Kedua sisi cukup berbagi app.signed-url.secret yang sama. Tidak ada ketergantungan ke sesi user, jadi aman untuk job/transaksi panjang yang berjalan di background.

Akses File

File adalah salah satu konsumen. Endpoint di SignedFileController.

GET /api/v1/files/signed/generate/{type}/{id} Mint URL — wajib login

type = temp | storage. Param opsional ttl (detik). Endpoint ini butuh JWT — hanya user terautentikasi yang boleh membuat signed URL.

# Response 200
{
  "data": {
    "url": "/api/v1/files/signed/storage/uuid?expires=1717003600&signature=aB3xK...",
    "expires_at": 1717003600,
    "signature": "aB3xK..."
  }
}
GET /api/v1/files/signed/{type}/{id} Serve file — publik, validasi signature

Membawa ?expires=&signature=. Divalidasi oleh interceptor global sebelum file dikirim. Tidak butuh JWT.

Contoh Frontend

// 1. Mint signed URL (pakai token JWT yang ada)
const res = await fetch(
  `/api/v1/files/signed/generate/storage/${'$'}{fileId}?ttl=600`,
  { headers: { Authorization: `Bearer ${'$'}{token}` } }
);
const { data } = await res.json();

// 2. Pakai langsung — tanpa header lagi
imgElement.src = data.url;

Interceptor Global

SignedUrlInterceptor terdaftar untuk semua /api/** (lihat WebConfig). Logikanya opt-in:

  • Request tanpa param signature → dilewatkan ke auth normal (JWT)
  • Request dengan signature → wajib valid, kalau tidak ditolak 401

Artinya endpoint apa pun bisa diakses via signed URL hanya dengan menyertakan ?expires=...&signature=... — validasinya terpusat di satu tempat.

⚠
Endpoint yang ingin diakses via signed URL tanpa JWT harus masuk permitAll di SecurityConfiguration (filter security jalan sebelum interceptor). /api/v1/files/** sudah di-permit.

Skema Signature

# Mode URL — canonical string
{path}:{expires}
# contoh: /api/v1/files/signed/storage/uuid:1717003600

# Mode Raw — payload apa adanya
{payload}

# Algoritma (sama untuk keduanya)
signature = BASE64URL( HMAC_SHA256(secret, data) )

Validasi

  1. (URL) Cek expires > now — kalau lewat tolak
  2. Hitung ulang HMAC dari data yang sama
  3. Bandingkan constant-time (MessageDigest.isEqual) — cegah timing attack
ℹ expires ikut ditandatangani, jadi client tidak bisa memperpanjang masa berlaku dengan mengubah nilai di URL — signature langsung tidak cocok.

Konfigurasi

app:
  signed-url:
    secret: ${'$'}{SIGNED_URL_SECRET:...}
    default-ttl-seconds: ${'$'}{SIGNED_URL_TTL_S:300}
Env VariableDefaultKeterangan
SIGNED_URL_SECRET(default dev)Secret HMAC bersama — wajib diganti di production, minimal 32 karakter acak. Service lain (Go) memakai secret yang sama.
SIGNED_URL_TTL_S300Masa berlaku default signed URL dalam detik

Catatan Keamanan

⚠
Secret adalah kunci kepercayaan. Jika bocor, siapa pun bisa membuat signed URL / signature yang valid. Simpan di environment variable, rotasi berkala, dan jangan commit ke repo.
AspekPenanganan
Tamper expires / path / payloadSemua ikut ditandatangani — tidak bisa diubah
Timing attackConstant-time comparison
Path traversal (file)File dicari by-UUID di DB, bukan dari path mentah
Mint URL sembaranganEndpoint generate butuh JWT — hanya user login yang bisa mint
Link bocorSet ttl pendek untuk resource sensitif (mis. 60 detik)
